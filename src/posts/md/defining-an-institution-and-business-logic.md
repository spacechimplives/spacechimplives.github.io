-  Code (revise to match turing machine)

 	A sequence of Instructions, communicated by Agent, executed by Executor

 	 Instruction:

 	(optional) a communication to Executor which provides context

 	- (required) a state of anticipation within Agent with a prepared communication channel / register physically capable of detecting a response signal

 	detection: a different resultant internal state based on the possible anticipated responses

 	- anticipation state:

 	a state in which something is raised out of equilibrium so that a controlled state change reaction might occur when a certain event happens







 	Agent

 	- something capable of giving instructions

 	must be able to transmit information

 	must be able to transmit different information based on internal states



 	must be able to receive information

 	- must be able to modify internal state differently in response to different incoming signals







 	Executor

 	- something capable of executing instructions

 	must be able to transmit information to agent's receptors





 	Special case equivalent to Turing Machine:

 	- Agent: head

 	detection: process of matching of state read from tape & head to table of instructions

 	- anticipation state: process of moving to new place on tape and reading state

 	- passes communication to Executor which provides context/token corresponding to expected behavior



 	Executor:

 	- whatever mechanism is moving along, reading from, and writing to tape

 	must successfully transmit information based on read to head



 	









 

 	- Institution

 	A set of people or groups whose behavior is coupled such that, within in a given context, their behavior is almost fully determined by that set

 	- Can be modeled by game theory

 	such that an environmental stimulus (a potential reward) can trigger the execution of a turing-complete program.



 	Turing machine requirements

 	- Acts as turing machine for given coroutine goals

 	Independent routines for each

 	make sale

 	- fulfill sale

 	- handle customer service





 	Halt State

 	- task is completed, ticket closed, order filled, sale made, etc.



 	Start State

 	- initial conditions are provided when starting a task, e.g. purchase order, inventory, contact information, price, etc. as well as business processes available to members



 	A finite set of symbols

 	- businesses generally work within a subset of the local language.  Usually this is also augmented with specified, stereotyped vocabulary for certain things



 	A finite set of rules or transition

 	- this is typified by the "suggestion box" that immediately dumps into the trash can or the common customer service experience of a theoretically possible request  rendered impossible due to lack of ability to transition to that state

 	shows constraint but not finite-ness



 	transitions between states are finite because business processes must be recorded.


 	A finite set of states

 	- everything in the institutions "grasp" is either recorded somewhere, or remembered by a member

 	for large institutions it is more obvious



 	inventory, customer lists, purchase orders, etc.

 	- ultimately transformable into human-parseable tokens





 	It seems like the idea that businesses engage in turing-complete logic isn't too controversial, I guess the controversial part of the claim is that's al l they do.

 	- Every state that an institution can parse is a state that is reachable through its business logic

 	- Similar to the fact that we cannot perceive anything that doesn't interact with our cells

 	We then cannot process any information from that perpection unless the cell sends some sort of signal that our body can process
